# Helm — Blue Dots Economy

Three umbrella charts that together stand up the full Blue Dots Economy
stack on a Kubernetes cluster (typically EKS, provisioned via `opentofu/`).

Install in this strict order:

| # | Chart             | Namespace         | What it deploys                                                  |
|---|-------------------|-------------------|------------------------------------------------------------------|
| 1 | `common-services` | `common-services` | ingress-nginx, cert-manager, `letsencrypt-prod` ClusterIssuer, shared Postgres + Redis |
| 2 | `signals`         | `signals`         | Signals monorepo: api, ui, notification-service, match-score     |
| 3 | `aggregator`      | `aggregator`      | Aggregator portal: web (BFF), api, worker, keycloak              |

Everything is driven by **`install.sh`** inside each environment directory
(`opentofu/aws/<env>/install.sh`) — one script for both cloud bootstrap and
helm deploy. There is no Makefile.

## Where the secrets come from

Each chart gets **one generated values file**, written at the YAML **root**
level so helm reads it directly via a single `-f` — no slicing, no `yq`:

```
opentofu/aws/<env>/common-services-values.yaml
opentofu/aws/<env>/signals-values.yaml
opentofu/aws/<env>/aggregator-values.yaml
```

They are generated by the `output-file` opentofu module
(`opentofu/aws/modules/output-file/`), which reads outputs from `network`,
`eks`, `iam`, `storage`, and `random_passwords` and templates them into one
`.tfpl` per chart. Shared secrets (e.g. the redis password, used by all
three) are templated into each file directly at the tofu layer — so the
files are self-contained, with no cross-file YAML anchors.

```yaml
# common-services-values.yaml
credentials:
  postgresAdminPassword: <generated>
  aggregatorPassword: <generated>
  dpgPassword: <generated>
  redisPassword: <generated>

# aggregator-values.yaml
global: { dataPlatform: { namespace: common-services, ... } }
secrets: { postgresPassword, kcBootstrapAdminPassword, ... }
api:    { serviceAccount: { create: true, annotations: { eks.amazonaws.com/role-arn: <arn> } } }
worker: { serviceAccount: { ... same ARN ... } }

# signals-values.yaml
credentials: { postgresql, redis }
api: { secrets: { ... }, ingress: { ... }, ... }
ui / notification-service / match-score: { ... }
```

Result: one source of truth per chart, no `--set` flags, no kubectl-secret
fetches, no `openssl rand` retries.

> The shared password values are duplicated across files where two charts
> need the same secret (templated from the same tofu variable), so they can
> never drift apart — a single `random_passwords` value feeds both.

## Ingress host names

Public FQDNs are **not** derived or hard-coded in the charts — they are read
verbatim from `opentofu/aws/<env>/global-values.yaml` and templated into the
generated values files:

| `global-values.yaml` key | Used as           | Lands in                              |
|--------------------------|-------------------|---------------------------------------|
| `signals_host`           | signals API host  | `signals-values.yaml` → `api.ingress.hosts[0].host` + TLS |
| `signals_ui_host`        | signals UI host   | `signals-values.yaml` → `ui.ingress.hosts[0].host` + TLS  |
| `aggregator_host`        | aggregator host   | `aggregator-values.yaml` → `global.publicHost` (api + web share it) |

```yaml
# opentofu/aws/<env>/global-values.yaml
global:
  signals_host:    "api.purpledots.servehalflife.com"   # signals API ingress
  signals_ui_host: "purpledots.servehalflife.com"       # signals UI ingress
  aggregator_host: "aggregator.servehalflife.com"       # aggregator ingress
```

Each value is used **as-is** (full FQDN — no `api.signals.` prefixing). Change
a host in `global-values.yaml`, re-run `terragrunt apply` on the `output-file`
unit to regenerate, then redeploy the affected release. Point DNS for these
hosts at the ingress-nginx LoadBalancer created by `common-services`.

## Prerequisites

- Kubernetes 1.24+ (EKS provisioned via `opentofu/aws/template`)
- `kubectl` with current-context pointed at the target cluster
- `helm` v3.12+
- `terragrunt` + `tofu` for generating the per-chart values files
- DNS A/CNAME records for the host names set in `global-values.yaml`
  (`signals_host`, `signals_ui_host`, `aggregator_host`), pointing at the
  LoadBalancer hostname created by ingress-nginx

(`yq` is **no longer required** — slicing is gone.)

## Quick start

```bash
# 1. Bootstrap a new environment from the template
cp -R opentofu/aws/template opentofu/aws/dev   # or staging/, prod/, ...
cd opentofu/aws/dev

# 2. (Optional) edit dev/global-values.yaml to override defaults

# 3. Provision cloud + generate the per-chart values files
#    (no-arg install.sh runs the full infra bootstrap)
bash install.sh
#    ...or just regenerate the values files:
#    terragrunt run --module-name output-file apply

# 4. Create namespaces + ghcr-pull secret, then install all 3 releases
GHCR_PAT=ghp_xxx bash install.sh deploy_all_services
```

`install.sh` auto-derives all paths from its own location, so no edits are
required when cloning template → dev/staging/prod.

`deploy_all_services` chains: `preflight → create_namespaces_and_secrets →
deploy_common_services → deploy_signals → deploy_aggregator`, each helm step
with a readiness `--wait`. `deploy_common_services` applies the gp3 default
StorageClass first (shared Postgres/Redis PVCs bind to it).

### Per-chart commands

```bash
bash install.sh deploy_common_services
bash install.sh deploy_signals
bash install.sh deploy_aggregator
```

### Image-pull secret only

```bash
GHCR_PAT=ghp_xxx bash install.sh create_namespaces_and_secrets   # PAT via env
bash install.sh create_namespaces_and_secrets                    # interactive prompt
```

### Checks (no install)

```bash
bash install.sh preflight   # tooling + cluster + values files present
bash install.sh lint        # helm lint all 3 charts
bash install.sh dry_run     # helm --dry-run all 3 against the current cluster
```

Chain any functions: `bash install.sh lint dry_run`.

## Override the generated file paths

If you keep the values files elsewhere (a separate secrets repo, manually
maintained files), point the script at them via env vars:

```bash
CS_VALUES=/etc/bluedots/common-services-values.yaml \
SIGNALS_VALUES=/etc/bluedots/signals-values.yaml \
AGG_VALUES=/etc/bluedots/aggregator-values.yaml \
  bash install.sh deploy_all_services
```

Namespaces and release names are likewise overridable: `CS_NS`, `SIGNALS_NS`,
`AGG_NS`, `CS_REL`, `SIGNALS_REL`, `AGG_REL`.

## Why this order

- **common-services first.** `ingress-nginx`, `cert-manager`, the shared
  Postgres StatefulSet and the shared Redis live here. Subsequent charts
  attach to them.
- **signals second.** Self-contained app stack. Uses common-services Postgres
  user `dpg` (password = `common-services.credentials.dpgPassword`) and the
  shared Redis.
- **aggregator last.** Same dependency on common-services; uses Postgres user
  `aggregator` (password = `common-services.credentials.aggregatorPassword`).
  Longer rollout because the Keycloak init Job runs after Postgres is Ready.

## Rotation

Secrets are stable across `tofu apply`s — `random_id` / `random_password`
resources persist in terraform state. To rotate:

```bash
cd opentofu/aws/<env>
terragrunt run --module-name random_passwords destroy   # or:
#   tofu taint -allow-missing random_id.<name>          (one secret)
terragrunt run --all apply                              # regenerates the 3 values files

bash install.sh deploy_all_services                     # re-runs all 3 helm releases
```

Because the same tofu variable is templated into every file that needs it
(e.g. `signalstack_admin_key` lands in both `aggregator.secrets.signalstackAdminKey`
**and** `signals.api.secrets.data.AGGREGATOR_DPG_API_KEY`), the releases
never drift apart.

## Aggregator overrides applied in this repo

`helm/aggregator/values.yaml` is patched vs. upstream:

```yaml
ingress-nginx:
  enabled: false        # common-services owns the controller cluster-wide
cert-manager:
  enabled: false        # common-services owns cert-manager + ClusterIssuer
```

The chart still vendors both subcharts under `charts/` (so it remains
installable standalone elsewhere), but they're inert here. The
`templates/clusterissuer.yaml` is gated on `cert-manager.enabled` and
therefore renders nothing.

If you ever want to run `aggregator` against a cluster that has *no*
`common-services` chart, flip both flags back to `true` and skip
`deploy_common_services`.

## Cleanup

```bash
bash install.sh cleanup_all_services    # reverse order: aggregator → signals → common-services
```

Individual:

```bash
bash install.sh destroy_aggregator
bash install.sh destroy_signals
bash install.sh destroy_common_services
```

Each `destroy_*` uninstalls the release and deletes its namespace (which
drops that namespace's PVCs) — **destructive** for the data charts.

To tear down the cloud infra as well:

```bash
bash install.sh destroy_tf_resources
```

## Troubleshooting

- **`ERROR: values file not found: <path>`** (from `preflight`) — opentofu
  hasn't generated the files yet. `cd opentofu/aws/<env>` then `terragrunt
  run --all apply` (or `terragrunt run --module-name output-file apply`).
  Point the script elsewhere via `CS_VALUES=... SIGNALS_VALUES=...
  AGG_VALUES=... bash install.sh ...` if the files live elsewhere.
- **`ERROR: cluster unreachable`** (from `preflight`) — your kubeconfig
  current-context isn't pointed at the target cluster. Check `kubectl config
  current-context`.
- **`rotate-ghcr-pull.sh missing`** — run from the env directory that
  contains it (`opentofu/aws/<env>`); `install.sh` resolves it next to
  itself.
- **PVCs stuck `Pending`** — gp3 isn't the default StorageClass. Run
  `bash install.sh apply_gp3_default_sc` (also run automatically by
  `deploy_common_services`).
