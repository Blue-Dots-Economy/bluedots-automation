# Helm — Blue Dots Economy

Three umbrella charts that together stand up the full Blue Dots Economy
stack on a Kubernetes cluster (typically EKS, provisioned via `opentofu/`).

Install in this strict order:

| # | Chart            | Namespace    | What it deploys                                                  |
|---|------------------|--------------|------------------------------------------------------------------|
| 1 | `platform`       | `common-services` | ingress-nginx, cert-manager, `letsencrypt-prod` ClusterIssuer, shared Postgres + Redis |
| 2 | `dpg`            | `dpg`        | DPG monorepo: api, ui, notification-service, match-score         |
| 3 | `aggregator-dpg` | `aggregator` | Aggregator portal: web (BFF), api, worker, keycloak              |

## Where the secrets come from

All cluster outputs **and** chart secrets live in **one** file:

```
opentofu/aws/template/global-cloud-values.yaml
```

It is generated by the `output-file` opentofu module, which reads outputs
from `network`, `eks`, `iam`, `storage`, and `random_passwords`. The file
contains a section per chart at the YAML root:

```yaml
global: { vpc_id, cluster_endpoint, app_sa_role_arn, ... }
default_passwords: { redis_password, signalstack_admin_key, ... }
common-services:
  credentials:
    postgresAdminPassword: <generated>
    aggregatorPassword: <generated>
    dpgPassword: <generated>
    redisPassword: <generated>
aggregator:
  secrets: { postgresPassword, kcBootstrapAdminPassword, ... }
  api: { serviceAccount: { create: true, annotations: { eks.amazonaws.com/role-arn: <arn> } } }
  worker: { serviceAccount: { ... same ARN ... } }
signals:
  credentials: { postgres, redis, api }
  api: { serviceAccount: { ... same ARN ... } }
```

Each chart's `make ...-install` target slices out its top-level wrapper via
`yq` into `.helm-slices/<chart>.yaml`, then feeds that slice to helm via
`-f`. Result: one source of truth, no `--set` flags, no kubectl-secret
fetches, no `openssl rand` retries.

## Prerequisites

- Kubernetes 1.24+ (EKS provisioned via `opentofu/aws/template`)
- `kubectl` with current-context pointed at the target cluster
- `helm` v3.12+
- `yq` v4 (mikefarah; **not** the python-yq)
- `terragrunt` + `tofu` for regenerating `global-cloud-values.yaml`
- DNS A/CNAME records for the host names referenced in chart `values.yaml`
  files, pointing at the LoadBalancer hostname created by ingress-nginx

## Quick start

```bash
# 1. Provision cloud + generate global-cloud-values.yaml
cd opentofu/aws/template
terragrunt run --all apply
cd -

# 2. Install all three helm releases in order
make install
```

`make install` chains `platform-install → dpg-install → aggregator-install`,
each with a readiness wait.

Per-chart targets:

```bash
make platform-install
make dpg-install
make aggregator-install
```

Re-slice + dry-run without applying:

```bash
make slices       # rebuilds .helm-slices/<chart>.yaml from global-cloud-values.yaml
make dry-run      # helm --dry-run for all 3 against the current cluster
```

Static checks (no cluster needed):

```bash
make lint
make template
```

`make help` lists every target.

## Override the generated file path

If you keep `global-cloud-values.yaml` elsewhere (e.g. a separate secrets
repo, an existing manually maintained file), pass `CLOUD_VALUES=...`:

```bash
make install CLOUD_VALUES=/etc/bluedots/global-cloud-values.yaml
```

## Why this order

- **platform first.** `ingress-nginx`, `cert-manager`, the shared Postgres
  StatefulSet and the shared Redis live here. Subsequent charts attach to
  them.
- **dpg second.** Self-contained app stack. Uses common-services Postgres
  user `dpg` (password = `common-services.credentials.dpgPassword`) and the
  shared Redis.
- **aggregator-dpg last.** Same dependency on platform; uses common-services
  Postgres user `aggregator` (password = `common-services.credentials.aggregatorPassword`).
  Longer rollout because the Keycloak init Job runs after Postgres is Ready.

## Rotation

Secrets are stable across `tofu apply`s — `random_id` / `random_password`
resources persist in terraform state. To rotate:

```bash
cd opentofu/aws/template
terragrunt run --module-name random_passwords destroy   # or:
#   tofu taint -allow-missing random_id.<name>          (one secret)
terragrunt run --all apply

cd -
make install      # re-runs all 3 helm releases with the new values
```

Because the same scalar value is anchored in `global-cloud-values.yaml`
(e.g. `signalstack_admin_key` aliased into both `aggregator.secrets.signalstackAdminKey`
**and** `signals.credentials.api.data.AGGREGATOR_DPG_API_KEY`), the two
helm releases never drift apart.

## Aggregator-dpg overrides applied in this repo

`helm/aggregator/values.yaml` is patched vs. upstream:

```yaml
ingress-nginx:
  enabled: false        # platform owns the controller cluster-wide
cert-manager:
  enabled: false        # platform owns cert-manager + ClusterIssuer
```

The chart still vendors both subcharts under `charts/` (so it remains
installable standalone elsewhere), but they're inert here. The
`templates/clusterissuer.yaml` is gated on `cert-manager.enabled` and
therefore renders nothing.

If you ever want to run `aggregator-dpg` against a cluster that has *no*
`platform` chart, flip both flags back to `true` and skip `make platform-install`.

## Cleanup

```bash
make uninstall          # reverse order: aggregator → dpg → platform
```

Individual:

```bash
make aggregator-uninstall
make dpg-cleanup        # also drops PVCs
make platform-uninstall
make slices-clean       # nuke the .helm-slices/ cache
```

`dpg-cleanup` is destructive — it deletes Postgres/Redis PVCs.

## Troubleshooting

- **`make` errors with `ERROR: <path> not found`** — opentofu hasn't been
  applied yet, or the path is wrong. Run `terragrunt run --all apply`
  under `opentofu/aws/template`, or set `CLOUD_VALUES=...` on the make
  command.
- **`ERROR: $(CLOUD_VALUES) has no top-level key "<chart>"`** — the
  generated file is from an older opentofu version that didn't emit the
  chart wrapper. Re-run `terragrunt run --module-name output-file apply`.
- **Wrong yq** — Python's `yq` does not support `.["common-services"]`
  syntax. Install the Go port: `snap install yq` or `brew install yq`.
